
# Lancache-Storage

Веб хранилище для файлов. Проксирует трафик и сохраняет файлы если это нужно. 
Например, у вас много устройств в одной сети, и чтобы каждый раз не скачивать обновление, игры, другие файлы с интернета.

Запускаете Lancache-Storage, и он сохраняет файлы в хранилище. Другое устройство будет скачивать уже из хранилища.

Это другая реализация https://lancache.net проекта.

### Основные проблемы в Lancache.net:
- По умолчанию стоит разбивка на 1мб - дополнительная нагрузка на диск. Выключить нельзя, можно только увеличить размер.
- Nginx cache не работает с 206 заголовками. Используется для скачки какого-то отрезка файла по диапазону через заголовок Range. 
В результате проблемы с такими файлами, приходиться убирать их из Lancache.net
- Не понятно что лежит в кэше.
- Работает через upstream - nginx отправляет запрос на себя же.

### Возможности Lancache-Storage:
- Файл сохраняется как он есть(без разбивки)
- Работает с 206 заголовком.
- Можно узнать что лежит в кэше.

## Как запустить:
### DNS сервер
Рекомендую использовать мой [Pi-Hole](https://github.com/DjoSmer/pi-hole-with-lancache-docker) где вы можете управлять через web интерфейс, что проксировать.
Перейдите по ссылке и там показано как запустить и настроить его.

Если вам не нужно управление, и чтобы работало сразу, можете запустить DNS сервер от [Lancache.net](https://lancache.net/docs/containers/dns/)

### Lancache-Storage
- Скачайте
  - *[docker-compose.yml](./docker-compose.yml)*
  - *[example.env](./example.env)* -> **.env**. 
- Отредактируйте **.env** под ваши настройки.
- **docker compose up -d**

### Docker Сервисы
- storage - Ядро проекта, проверяет, загружает, сохраняет файлы.
- storage-watch - Следит за хранилищем. Проверяет чтобы хранилище не вышло за STORAGE_DISK_SIZE и удаляет файлы если их время >= STORAGE_MAX_AGE.
- storage-sync - Анализирует хранилище, сверяет что есть в базе данных и в хранилище. Если в хранилище есть файлы, которых нету в базе данных, то добавляем из базу данных. 
Его можно настроить на запуск 1 раз в месяц.

## Как работает: 
Запрос приходит на Nginx. Nginx отправляет запрос на Storage, проверяет есть ли такой файл.
- Если есть → возвращает 302 заголовок и путь, где лежит файл. Nginx подхватывает заголовок и загрузка файла идет из хранилища.
- Если файла нет → возвращается 404 заголовок. Nginx подхватывает заголовок и загрузка идет напрямую. Storage скачивает файл и добавляет его в хранилище.

## Мой опыт:
- Если у lancache сервера(ЛК) 1G сетевая карта - то хватит одного ссд, т.к. узкое горлышко сетевая карта.
- Если ЛК 10G - 1 ссд не хватит, нужен уже несколько. Я использовал TrueNas и 4 ssd.

### Немного теории:
Скорость SSD действительно высокая, но для серверов важно понимать разницу между **IOPS** и **пропускной способностью (MB/s)**.

**Примерные характеристики:**

- HDD: ~100–300 IOPS, ~100–250 MB/s
- SATA SSD: ~50,000–100,000+ IOPS, ~500–560 MB/s
- NVMe SSD: ~500,000–1,000,000+ IOPS, 3,000+ MB/s

**IOPS (Input/Output Operations Per Second)** — это количество операций ввода-вывода в секунду.

### Размер блока и ZFS (TrueNAS)
Во многих файловых системах используется блок 4 KB.
Но в ZFS (например, в TrueNAS) используется параметр **recordsize**, который по умолчанию равен **128 KB**.

Это означает:
-файл разбивается на блоки **до 128 KB**
-для больших файлов используется крупный блок → меньше операций → выше эффективность

Пример, файл 10 MB:
- при 4 KB → ~2560 блоков
- при 128 KB → ~80 блоков

Большой recordsize (128K):
- уменьшает накладные расходы
- снижает количество операций
- увеличивает throughput при последовательном доступе

---

10G сеть = примерно **1.25 GB/s**

- **один HDD** → максимум ~0.2 GB/s → узкое место
- **один SATA SSD** → ~0.5 GB/s → тоже не раскрывает 10G
- только **NVMe** способен приблизиться к лимиту, но очень дорого и вся скорость пропадает когда диск заполняется на 80%, а у ЛК диски могут заполняться и на 90%.

Чтобы утилизировать 10G сеть, делают:
- RAID0 / ZFS stripe (несколько дисков)
- RAIDZ / mirror (с балансом скорости и надежности)

Поэтому можно взять:
- 6 × HDD → ~1.2 GB/s (почти насыщает 10G), но я бы взял 7-8 на запас
- 4 × SATA SSD → ~2 GB/s - их и выбрали, надежность данных не требуется.
